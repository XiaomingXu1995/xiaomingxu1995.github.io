## debug 
在debug的过程中积累的一些小经验。**遇事不决PRINT**

### 有关代码测试：
* 代码测试的时候要模块化，修改了哪一部分就马上记得去测试修改部分的正确性。不要等到最后整体测试，一旦出错，不容易定位出错位置。

### 重定向输出

Linux 标准文件描述符：

|id  |description|explaination|覆盖原文件|追加到文件|
|:---|:----------|:-----------|:--------|:--------|
|0   |stdin      |标准输入    |`>`      |`>>`     |
|1   |stdout     |标准输出    |`>`      |`>>`     |
|2   |stderr     |标准错误    |`>`      |`>>`     |

   所以在文件重定向的时候，将标准错误输出重定向到文件：
  ``` 
  ./a.out 2>result.err
  ```
   如果同时重定向标准输出和错误输出：
   ```
   ./a.out 1>result.out 2>result.err
   ```
   如果将`stdout`和`stderr`重定向到同一个文件中：
   ```
   ./a.out &>result
   ```
   不过`bash shell`可能会赋予错误消息更高的优先级，错误消息会优先显示，避免错误信息散落在输出文件中。

### 在某个固定位置退出
设置程序执行到具体位置退出，可以透过`return 返回值类型`或者`exit(0)`来实现，用`return`比`exit`要规范一些，因为`exit`是强制退出，很多申请的内存并没有释放，某些析构函数没有得到执行。`exit()`函数需要`#include <cstdlib>`。

### 活用Makefile
曾经在优化一个比较大的项目的时候，为了获得某个文件编译之后的汇编文件，需要对这个文件进行单独的编译，得到汇编文件（在编译的时候加上`-S`选项）。但是对于编译的其他参数又不能良好的把握，这个时候就要利用`Makefile`的时间戳特性，可以将需要编译的文件进行一个“无伤大雅”的修改（比如在某一行最后加一个空格），然后`make`,该文件最新修改，就可以得到编译该文件的具体命令，复制出来加上`-S`参数就可以进行具体的编译汇编指令了。（当然`touch`命令也刷新文件的时间戳）。

### 灵活的跟踪定位BUG
debug过程中一般分为两个比较大的步骤，一个是解决编译问题：比如语法错误等，在编译过程中就可以检测出来。另外一个就是运行时错误：可以编译出可执行文件，但是在运行时程序非正常终止，比较常见的是segment fault，很多情况下是访问非法地址，例如数组越界，或者在传递参数的过程中指针传递错误。前者可以自己检测数组申请内存大小和访问数组的索引是否越界，后者可以跟着程序判断参数的传递是否真实有效，要合理利用printf准确定位bug位置，`printf("%p\n",point)` 打印出具体地址的值，判断地址传递过程中是否会有问题，尤其是在类的成员变量传递过程中会出现相应的问题。
